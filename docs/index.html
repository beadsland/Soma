S.cast (value)
Given a Lua value, cast as an immutable Soma term.

STATUS: Top-down cast logic only. Underlying target types have yet to be implemented.

We do our best to render any Lua data type thrown at us as the nearest Elixir-style equivalent, as follows:

Lau dataSoma term
nil atom(':undef')
boolean, true atom(':true')
boolean, false atom(':false')
number, NaN atom(':nan')
number, infinity atom(':inf')
number, negative infinity atom(':-inf')
number, sans mantissa integer
number, otherwise float
string, matching /^:/ atom
string, matching /^':.*'$/ atom
string, otherwise string
userdata error
function function
thread pid (if known to scheduler, otherwise error)
table, has __call metamethod function
table, an array list
table, empty error
table, otherwise map

Various exceptional type values afforded by Lua reduce to atoms in Soma. The userdata type is a beast peculiar to Lua's C API; we don't even try to cast it. Only those coroutine threads spawned within the Soma scheduler architecture will cast to an associated PID. (Mixing and matching roll-your-own coroutines with the Soma scheduler is not recommended.)

Arrays and other non-empty tables readily translate to Soma lists and maps, with the __call metamethod presenting a special case. That said, no provision is made within Soma for the casting, as terms, of those Lua tables with more intensive behind-the-curtain jiggery-pokery. (But then, if your problem domain calls heavily on the semantics of Lua's highly extensible tables, it probably isn't something you're want to be doing in Soma anyway.)

Parameters:

  • value Any Lua expression, excluding userdata and empty tables.

Returns:

    A term class table representing an immutable Soma term.
